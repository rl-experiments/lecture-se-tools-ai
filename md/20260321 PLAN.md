### Block 4.1 ‚Äî Prompt Engineering for Developers (2h, 08:00‚Äì10:00) ‚Äî Theory + Practice

**Format:** `SLIDES` (45min) ‚Üí `EXERCISE` (1.25h)

This is the most transferable skill in the course. Every developer will use this daily.

**SLIDES (45min):**
- Why prompt quality matters: same model, 10√ó different output quality
- **Explicit criteria > vague instructions:**
  - Bad: "check that comments are accurate"
  - Good: "flag comments only when claimed behavior contradicts actual code behavior"
  - Bad: "be conservative" ‚Üí model ignores this
  - Good: "report only: null dereference, resource leaks, SQL injection. Skip: naming, formatting."
- **Few-shot examples (the single most powerful technique):**
  - 2‚Äì3 examples eliminate ambiguity better than paragraphs of instructions
  - Show input ‚Üí expected output for ambiguous cases
  - Model generalizes from examples to novel patterns
- **Structured output:**
  - `tool_use` + JSON schemas = guaranteed format, no parsing errors
  - `tool_choice`: `"auto"` vs `"any"` vs forced tool selection
  - Schema design: required vs optional fields, nullable to prevent hallucination
- **The interview pattern:**
  - Instead of: "implement caching for my API"
  - Better: "Before implementing, ask me questions about: invalidation strategy, failure modes, expected load"
  - Claude asks ‚Üí student thinks ‚Üí better design before any code

**EXERCISE (1.25h ‚Äî 6 students, lots of hands-on):**
- **Exercise 1: Prompt comparison (30min)**
  - All 6 students write a prompt for the same task: "Add input validation to WeatherController"
  - Compare outputs: which prompt produced the best code? Why?
  - Iterate: refine the worst prompt using explicit criteria + few-shot examples
- **Exercise 2: Iterative refinement (30min)**
  - Instructor provides a pre-written failing test (e.g., `WeatherParserTest` with 3 assertions)
  - Students give Claude Code the test file ‚Üí "Make this test pass by implementing WeatherParser"
  - When implementation has issues ‚Üí share the test failure output back ‚Üí Claude iterates
  - Goal: experience one full cycle of test-driven development with an LLM. One good cycle teaches more than three rushed ones.
- **Exercise 3: Structured review output (15min)**
  - Write a prompt that produces JSON-formatted code review findings
  - Run on each other's code (3 pairs) ‚Üí compare finding quality

### Block 4.4 ‚Äî MCP + Agentic Architecture + cognitive-core Agents (2h, 13:30‚Äì15:30) ‚Äî Theory + Practice

**SLIDES + LIVE DEMO ‚Äî MCP: Model Context Protocol (25min, 13:35‚Äì14:00):**
- What is MCP? The standard for connecting LLMs to external tools and data
  - Tools = actions (call an API, query a database)
  - Resources = data catalogs (list issues, browse docs)
  - Servers = backends that expose tools + resources
- `.mcp.json`: project-level (shared, in Git) vs `~/.claude.json` (personal)
- Environment variable expansion: `${GITHUB_TOKEN}` ‚Äî no secrets in Git
- **Tool descriptions are everything:**
  - Bad: "Retrieves customer information" ‚Äî too vague, model picks wrong tool
  - Good: input formats, example queries, boundaries, when NOT to use it
- Principle: 4‚Äì5 tools per agent, not 18. More tools = worse selection.
- **Instructor demo:** Connect Claude Code to GitHub MCP server, show tool discovery, run an ambiguous request ‚Äî does it pick the right tool?
- (Students can connect MCP independently if interested ‚Äî not required for grading)

**SLIDES ‚Äî Agentic Architecture (25min, 14:00‚Äì14:25):**
- The agentic loop: request ‚Üí `stop_reason` check ‚Üí tool execution ‚Üí next iteration
  - `"tool_use"` = continue calling tools, `"end_turn"` = done
- Coordinator-subagent pattern (hub-and-spoke):
  - Coordinator decomposes task, delegates to specialists, aggregates results
  - Subagents have isolated context ‚Äî no automatic inheritance, context must be passed explicitly
- Agent SDK hooks: `PostToolUse` for data normalization, tool call interception for compliance enforcement
- Task decomposition: fixed pipelines (prompt chaining) vs adaptive investigation
- **Context management (woven in, not a separate block):**
  - Context window limits are real ‚Äî models lose information in the middle of long inputs
  - Scratchpad files: persist findings across context boundaries
  - `/compact`: reduce context when sessions get long
  - Subagent delegation: isolate verbose exploration, return summaries
  - Session resumption: `--resume` for continuing work across sessions
- Error propagation: structured errors (failure type + partial results + what was attempted), not generic "failed"

**SLIDES ‚Äî cognitive-core as Architecture Case Study (15min, 14:25‚Äì14:40):**
- How hooks, skills, and agents compose into a layered quality system:
  - Hooks = reflexes (automatic, no student action)
  - Skills = tools (student initiates, interprets result)
  - Agents = review partners (student interacts, agent reasons)
- `@code-standards-reviewer` agent: definition, system prompt, tool restrictions
- Fitness gates = programmatic enforcement (deterministic, guaranteed) vs prompt instructions (probabilistic, hope-based)
- Student Lab Pack vs full enterprise cognitive-core: students use 9% of skills, 10% of agents

**EXERCISE ‚Äî cognitive-core Agent: Intelligent Review Partner (50min, 14:40‚Äì15:30):**
- Activate `@code-standards-reviewer` agent
- Each student: agent reviews their Weather App
  - Compliance summary, specific violations, architecture compliance
  - Each finding includes provenance: documented / verified / inferred / automated
  - Students learn: where do standards come from? What's a rule vs a suggestion?
- **Key rule: the agent does NOT fix your code. You read. You understand. You fix.**
- Compare across the session:
  - Session 2: manual `mvn checkstyle:check` ‚Üí you find violations

## Risk Mitigation

| Risk | Mitigation |
|------|------------|
| Claude Code installation fails | Install instructions sent 2 weeks before Session 3. Buffer time in Block 3.2. 6 students = instructor can troubleshoot individually. |
| Student has no Claude Pro license | Pair programming with licensed student. Instructor can demo. |

| Students leave everything to the last week | Grading criteria communicated from Session 1. `/fitness` score requires sustained work, not a last-minute sprint. |
| MCP demo fails in class | MCP is instructor demo only, not required for grading. No dependency. |
